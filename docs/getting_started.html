Universal Auto Recorder & Playback

Getting-started guide · FTC SDK 11.0.0 · drive your autonomous once, replay it in Auto

What this is

A record-and-replay autonomous tool. While you drive in TeleOp it records every motor and servo at 50 Hz; an Autonomous OpMode replays them with smooth interpolation. Zero config — it auto-discovers your hardware, so there are no motor names to edit. This guide walks the integration in a few numbered steps and is honest about where the approach works well and where it doesn't. Full reference lives in the universal/readme.md.

Faithful by control mode: a closed-loop velocity motor (a flywheel on RUN_USING_ENCODER + setVelocity) is recorded and replayed as an RPM target, not raw power — so it holds speed regardless of battery voltage. Open-loop power channels are additionally voltage compensated on playback.

0 · Prerequisites

  • FTC SDK 11.0.0 project (Android Studio or OnBotJava). No Pedro / Road Runner / dashboard needed.
  • Your recording OpMode is a LinearOpMode — waitForSlotSelection() blocks, so an iterative OpMode (init()/loop()) isn't supported as-is. Playback is already a LinearOpMode.
  • A working robot configuration on the Driver Station and a gamepad.
  • A consistent start position. Replay has no localization — decide now how you'll place the robot the same way every match (field wall, a start jig, tape marks).
  • A charged, consistent battery. Speeds depend on voltage; record on a fresh pack.

1 · Add the two files

1Copy into your project one time

What you do: drop UniversalAutoRecorder.java and UniversalAutoPlayback.java into TeamCode/…/universal/. Optionally add SampleRecorderTeleOp.java to try it without touching your own TeleOp.

That's the whole install. Nothing to configure.

The OpModes ship disabled. UniversalAutoPlayback and SampleRecorderTeleOp carry an @Disabled annotation so they stay off your OpMode list. Delete that one line in a file to make it appear on the Driver Station. (UniversalAutoRecorder is a plain utility class — no annotation, always available.)

2 · Wire the recorder into a TeleOp

2Add the recorder recorder

What you do: construct the recorder after your hardware/subsystems are set up, call capture() every loop, and close() on stop.

import org.firstinspires.ftc.teamcode.universal.UniversalAutoRecorder;

public class MyTeleOp extends LinearOpMode {
    private UniversalAutoRecorder recorder;

    @Override public void runOpMode() {
        // ... hardwareMap.get(...) every motor/servo AND set their run modes ...

        recorder = new UniversalAutoRecorder(hardwareMap, gamepad1, telemetry, this);
        recorder.waitForSlotSelection();   // slot-picker UI; before waitForStart()
        waitForStart();

        try {
            while (opModeIsActive()) {
                // ... your normal drive / mechanism code ...
                recorder.capture();        // every loop (self-limits to 50 Hz)
            }
        } finally {
            recorder.close();              // saves the recording (finally = saved even on a crash)
        }
    }
}

On the Driver Station: D-Pad ▲▼ select slot · B delete · A or ▶ PLAY begin.

Construct it anywhere in init. The recorder re-samples each motor's control mode when recording starts (your first capture()), so a flywheel only needs to be in RUN_USING_ENCODER by the time you press play and drive — not when you build the recorder. The V:/M: shown on the slot screen is just a best-effort hint.
Pass this (the 4-arg constructor). It lets STOP exit the slot picker cleanly and lets the Driver Station PLAY button confirm. The 3-arg constructor lacks this — prefer the 4-arg form shown above.
Call capture() every loop. Don't gate it behind a button — it rate-limits itself internally.

3 · Record your autonomous

3Drive it by hand TeleOp

What you do: place the robot at your start pose, run the TeleOp, pick an empty slot, press A, then drive the autonomous routine exactly as you want it replayed. Press ■ STOP when done — it saves automatically.

Tip: keep early routines short. A clean 8-second auto replays far more reliably than a sprawling one.

4 · Replay in Autonomous

4Run "Universal Auto Playback" Autonomous

What you do: select the Universal Auto Playback OpMode, pick the same slot with D-Pad ▲▼, optionally toggle voltage comp with X, then press ▶ PLAY.

It rebuilds every recorded device by name, restores directions / zero-power behavior / velocity PIDF, and replays with up to ~200 Hz interpolation. It stops at the end of the recording (or a safety timeout of the recording's length + 5 s, minimum 30 s).

Same start pose, every time. There is no localization. A few centimeters or degrees of offset at the start carries through the entire run. Align against the field wall or a jig before every replay.

5 · Troubleshooting

SymptomLikely causeFix
"No devices discovered"Config not active / wrong namesActivate the correct robot configuration.
Ends in the wrong spotBattery, start pose, wheel slipAlign the same way; keep voltage comp ON; re-record on a fresh pack.
Flywheel speed wrongNot in RUN_USING_ENCODER when recording startedPut the flywheel in velocity mode before you press play and drive.
Playback does nothingEmpty slot / column mismatchRe-record; confirm the slot shows a KB size.
"Device not found"Playback config differs from recording configMatch device names between robots.
Stops earlySafety timeoutTimeout is recording length + 5 s (min 30 s) — long autos are fine.

6 · When to use something else

This is output replay, not navigation. Reach for an odometry path-follower (Pedro, Road Runner) instead when you need centimeter-level accuracy, sensor/AprilTag reactions, or long precise multi-segment routines. Universal Record & Playback shines when you want a working auto fast, have no odometry, or are sharing an auto with an alliance partner — and as a reliable fallback.

Fidelity by device. Faithful: motor-power drivetrains, closed-loop velocity flywheels, servos, CR-servos. Caveats: field-centric drive needs the same start heading; open-loop lifts drift. Not reproduced: RUN_TO_POSITION arms (won't hit presets — drive them as velocity/power), a motor that changes mode mid-run, and iterative OpModes (recording is LinearOpMode-only).
