How the Universal Auto Recorder works

Diagram & explanation · FTC SDK 11.0.0 · “Drive your autonomous once, replay it”

In one sentence

It records what your motors and servos do while you drive the robot by hand in TeleOp — and replays exactly those outputs on a timeline during Autonomous. No path, no odometry, no configuration: it discovers your hardware itself.

The trick is deliberately simple. Instead of computing where the robot should go (like Road Runner or Pedro with odometry), the system just remembers which commands each device received at each moment. A recording is essentially a table: one row every 20 milliseconds, one column per device. On playback that table is “driven” again, smoothly interpolating between the rows.

The diagram — three phases

One row per 20 ms; one column per device. The number of columns adapts automatically to your hardware.
Phase 1 · Record

In TeleOp, capture() polls every discovered motor/servo 50 times per second and writes a row into a buffer. It reads the output (power, velocity, servo position) — not where the robot is.

Phase 2 · File

On stop, close() writes everything to slot_NN.csv. The header is self-describing: which devices, which type, which control mode. So playback needs no hard-coded names.

Phase 3 · Playback

The Autonomous OpMode reads the header, rebuilds exactly those devices, and drives the timeline — with linear interpolation for smooth motion.

Why “universal”?

Most auto tools want to know how your robot is built: which motors, which names, which gearing. This tool instead queries the HardwareMap at startup and simply takes everything it finds:

Discovery happens in the constructor; the recording's column count comes purely from the hardware found.

Faithful to control mode

A raw power value isn't the right quantity for every device. So the system records each motor the way it is controlled — and plays it back the same way:

Device / modeWhat's recordedPlayback
Power motor (RUN_WITHOUT_ENCODER)the commanded power value (−1…1)setPower(), plus battery-voltage compensated
Velocity motor (RUN_USING_ENCODER, e.g. a flywheel)the measured speed in ticks/ssetVelocity() with the original PIDF → holds the speed regardless of voltage
Servothe position (0…1)setPosition()
CR servothe power value (−1…1)setPower()
Battery compensation: open-loop power channels are scaled per frame by recorded voltage ÷ current voltage on playback. A half-empty battery on game day therefore doesn't undershoot. Velocity channels don't need it — the closed loop holds the speed anyway.

Smooth playback — interpolation

A sample point is only recorded every 20 ms. Outputting those steps directly would make the motion jerky. Playback therefore runs much faster (~200 Hz) and computes the in-between value linearly between two sample points — matched to the time actually elapsed, not the frame index:

What it deliberately can't do

Staying honest is part of the design — this is output replay, not a navigation plan:

  • No localization. It never knows where the robot really is. It does exactly what it did during the recording — whether or not the robot is in the right place. That's why an identical start position is mandatory.
  • Battery voltage matters. Compensation helps, but it can't push past 100 % saturation. Record on a fresh battery.
  • Open-loop drift accumulates. Carpet vs. tile, wheel wear — small errors add up over long autos. Short routines are more reliable.
  • RUN_TO_POSITION arms do not reach their presets (only the speed cap is captured). Drive such mechanisms as velocity/power instead.
  • No sensor behavior. It doesn't react to missed pickups, AprilTags, or defense.
Rule of thumb: use Record & Playback when you want a working auto fast, have no odometry, or are sharing an auto with an alliance partner. For centimeter-accurate, long, sensor-dependent routines a path follower (Pedro / Road Runner) is the better choice. The two combine well — Record & Playback as a fast start or fallback.
